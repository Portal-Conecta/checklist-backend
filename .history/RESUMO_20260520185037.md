# Resumo da Implementação: Camada de Segurança e Autorização (Checklist API)

Foi implementada com sucesso a autorização baseada nos perfis oficiais do Hub Conecta e na validação do token JWT.

---

## 🛠 O que foi feito

### 1. Dependências do Projeto (`pom.xml`)
- Adicionadas as dependências da biblioteca **JJWT (0.12.6)** para parsing e validação segura de tokens JWT (HMAC-SHA256).
- Adicionado o **spring-boot-starter-test** para suporte completo aos testes de integração MVC (`MockMvc` e autoconfigurações).

### 2. Infraestrutura de Segurança e Filtro JWT
- **`PortalUserPrincipal`**: Record que representa o usuário autenticado contendo ID do usuário, nome, perfis (`profiles`), vínculo de turma (`turmaId` para REPRESENTANTE), turmas vinculadas (`linkedTurmaIds` para DOCENTE) e escopo organizacional (`scope` para SENAI/WEG).
- **`JwtTokenProvider`**: Valida a assinatura HMAC-SHA256 do token e extrai os claims necessários mapeando-os para o principal.
- **`JwtAuthenticationFilter`**: Filtro HTTP que intercepta as requisições (`Authorization: Bearer <token>`), valida o token e popula o contexto de segurança do Spring.
- **`SecurityConfig`**: Configura a cadeia de filtros de segurança statless, habilita segurança por método (`@EnableMethodSecurity`), libera caminhos públicos (Swagger, Actuator) e integra tratadores de erro customizados.

### 3. Tratamento de Erros Padronizado (`ErrorResponseDTO`)
- **`CustomAuthenticationEntryPoint`**: Retorna status `401 Unauthorized` formatado como JSON via `ErrorResponseDTO` quando o token é inválido ou ausente.
- **`CustomAccessDeniedHandler`**: Retorna status `403 Forbidden` no formato `ErrorResponseDTO` quando a requisição é interceptada no nível do filtro HTTP.
- **`GlobalHandlerException`**: Adicionado tratamento para a exceção `@PreAuthorize` (`AccessDeniedException`) retornando `403 Forbidden` formatado no padrão do projeto.

### 4. Centralização de Regras de Autorização
- **`AuthorizationService`**: Centraliza as regras operacionais solicitadas:
  - `hasOperationalAccess()`: Bloqueia qualquer acesso de perfil `APRENDIZ`.
  - `canCreateExecution(Long turmaId)`: Permite criação para `REPRESENTANTE` (se a turma for a dele), `DOCENTE` (se estiver vinculada) ou `ADMINISTRADOR`.
  - `canViewDashboard()`: Permite acesso para `PERFIL_SENAI`, `PERFIL_WEG` e `ADMINISTRADOR`.
  - `canEditCompletedChecklist(String scope)`: Restringe a edição do checklist com base no escopo (`PERFIL_SENAI` edita "SENAI", `PERFIL_WEG` edita "WEG", `ADMINISTRADOR` tem acesso total).

### 5. Controlador de Testes (`DummyChecklistController`)
- Criado em `module/checklist/presentation` com as anotações `@PreAuthorize` adequadas para demonstrar as regras de acesso em endpoints de teste (`/executions`, `/dashboard`, `/completed` e `/operational`).

## 🧪 Validação dos Cenários (Testes)

Para garantir que toda a lógica funciona exatamente como especificado, foram criados dois conjuntos de testes automatizados:

1. **Testes Unitários (`AuthorizationServiceTest`)**:
   - 8 cenários que cobrem todas as variações de regras do `AuthorizationService` para os perfis `REPRESENTANTE`, `DOCENTE`, `PERFIL_SENAI`, `PERFIL_WEG`, `APRENDIZ` e `ADMINISTRADOR`.
2. **Testes de Integração (`SecurityIntegrationTest`)**:
   - 6 cenários simulando requisições HTTP reais utilizando `MockMvc` com tokens JWT assinados de mentira. Valida o comportamento do filtro de segurança, das anotações de método e a resposta padronizada JSON (401/403).
